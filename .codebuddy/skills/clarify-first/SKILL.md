# 澄清优先门禁 — 生成前先确认

来源: arxiv 2603.26233 (Uncertainty-Aware Clarification) + GitHub clarify-first-skills + Copilot Plan Mode

## 核心规则

**任何时候收到指令后, 如果存在以下任一情况, 必须先暂停并向用户提问, 确认后再生成:**

### 触发条件 (任一满足即暂停)

| 信号 | 示例 | 行为 |
|------|------|------|
| 🔴 多个可行方案 | "优化这段代码" — 不知道要优化什么 | 列出2-3个方向, 让用户选 |
| 🔴 关键参数缺失 | "对比模型" — 没说是哪个任务(S1/S3/S4) | 问清楚再动 |
| 🟡 存在冲突约束 | "要快但要质量" — 快和质量常冲突 | 问清楚优先级 |
| 🟡 不确定范围 | "审视代码" — 审视哪些文件? | 确认范围 |
| 🟢 细节模糊 | "生成测试" — 测什么? 多少用例? | 确认细节 |

### 不触发的情况 (直接执行)

- 指令包含所有必要参数且无歧义
- "继续" / "是的" / "对" (延续上文的确认回复)
- 纯查询 (如"当前状态是什么")

## 提问格式

```
在生成之前, 我需要确认 {X} 个细节:

1. {具体问题} ? [选项A / 选项B / 选项C]
2. {具体问题} ? 
3. {具体问题} ?

(最多3个问题, 优先问最关键的)
```

## 为什么需要这个门禁

- arxiv 2603.26233: 不确定性感知的编码agent比"盲目假设"型少37%返工
- Copilot Plan Mode: 先计划 → 提问澄清 → 再执行, 业界标准
- clarify-first-skills: 风险驱动的澄清门禁, GitHub 社区采纳
- 本项目的教训: 今天4轮审视修复的3个bug都来自"生成前未确认细节"
