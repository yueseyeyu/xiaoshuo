## 逐条审查

### [#1] 4层fallback章节提取是否覆盖所有格式？
**判定**: ✅
**依据**: `rhythm_analyzer.py:146-183` — 4层fallback: 第X章→序章+章X→数字标题→裸数字
**评价**: 覆盖了95%+网文格式。已知盲区: "Volume 1"等英文分卷、"番外"/"楔子"等特殊章名。
**建议**: 添加 `re.split(r"(番外|楔子|尾声|后记)")` 作为第5层fallback。

### [#2] 50字过滤是否丢有效内容？
**判定**: ⚠️
**依据**: `rhythm_analyzer.py:193` — `if len(pure_text) < 50: continue`
**问题**: 序章/Prologue有时<50字有效内容(如纯对话短章)，会被丢弃。
**建议**: `raise ValueError` 改为 `warn + skip`，或降到30字下限。

### [#3] PLEASURE_BOND 混入智斗/系统流特征？
**判定**: ❌
**依据**: `rhythm_analyzer.py:45` — `"面板弹出|推演显示|模拟得出|演算|第一条规则|从未见过"` 在 PLEASURE_COGNITIVE 中
**问题**: 这些词在 PLEASURE_COGNITIVE，不在 PLEASURE_BOND。但用户问的是 PLEASURE_BOND。让我重新确认...
**重新审查**: PLEASURE_BOND(L37-41) 包含 `"不放手|伸出手|碰了碰|贴着|靠着|扶住|接住|挡在|护在"` — 这些是肢体接触描述，在战斗场景中也会出现（"挡在身前""扶住战友"），可能被误判为羁绊。但消歧(L264)需要共现情感词/人称代词，有一定过滤效果。
**建议**: 评估战斗场景下的bond误判率，必要时添加排除规则（如前后20字检测冲突词）。

### [#4] 羁绊消歧硬编码主角名
**判定**: ❌ — 需立即修复
**依据**: `rhythm_analyzer.py:264` — `r"你|我|他|她|眼中|心里|轻声|沉默|握住|凝视|月|林默"`
**问题**: "月"和"林默"是《末日模拟器》的主角名，硬编码在通用分析引擎中。对其他10本火书，这些字符无意义且造成不对称处理。
**建议**: 将主角名外提为 config.yaml 中的 `author.character_names: ["月", "林默"]`，代码动态读取。对其他火书传空列表即可。

### [#5] 9类爽点等权聚合忽视r值
**判定**: ❌ — 需修复
**依据**: `rhythm_analyzer.py:267-270` — 所有count简单相加
**问题**: sacrifice_count(r=-0.13, 负相关)、cognitive_count(r=0.017, 近零) 等权混入会稀释 pos_density(r=0.445) 的信号。按 calibrate_v2 数据，带权聚合可提升 r vs LLM。
**建议**: 
```python
total_pleasure = (
    slap_count * 0.445 + level_count * 0.086 + crush_count * 0.30 +
    comeback_count * 0.178 + bond_count * 0.155 +
    cognitive_count * 0.017 + sacrifice_count * 0.053
) / sum_of_weights
```

### [#6] CONFLICT_KW_ALL 中"你敢|休想|去死"过匹配？
**判定**: ⚠️
**依据**: `rhythm_analyzer.py:86` — `"你敢|休想|去死|找死|不可能！|我不信|凭什么"`
**问题**: "不可能！"在网文中极高频（"这不可能！怎么可能！"），出现在日常对话中远多于生死冲突。同样"凭什么"在辩论/日常场景常见。
**评估**: 在10本末世火书中，战斗场景占比高，过匹配影响有限。但如果分析都市/言情类，冲突密度会被严重高估。
**建议**: 将高频通用词降权或限制在战斗上下文(前后30字含战斗词才计数)。

### [#7] conflict_density 的"密度"定义
**判定**: ⚠️ — 语义偏差
**依据**: `rhythm_analyzer.py:284-285` — `conflict_count / max(wc, 1) * 100`
**问题**: 这是"每百字冲突词次数"，不是真正的密度。短章节(1500字)天然得分高于长章节(5000字)，因为分母小。但章节字数本身是结构特征，不应影响冲突感知。
**建议**: 用 `conflict_count / chapter_count` 或按字数分组后内部排名。

### [#8] dialogue_ratio 权重2 vs r=-0.091
**判定**: ❌ — 噪音放大器
**依据**: `rhythm_analyzer.py:336` — `dialogue_ratio * 2`
**calibrate_v2 数据**: `dialogue_ratio: r_vs_llm_intensity = -0.091`
**问题**: 负相关意味着对话越多，LLM评分越低。但权重2意味着每个对话章多贡献~0.6分——这对于纯对话章(ratio=0.3)是+0.6，而LLM认为对话多的章反而更弱。这是明确的反向信号。注释说"从8降到2"，但2仍然太高。
**建议**: 权重降到0.5或直接剔除（同 level_count/sacrifice_count 处理）。

### [#9] 压缩公式 *0.7+1.5 的合理性
**判定**: ⚠️
**依据**: `rhythm_analyzer.py:343` — `pleasure_raw = pleasure_raw * 0.7 + 1.5`
**问题**: 1.5偏移使所有章强度≥1.5，压缩了低分区([0,1.5]变[1.5,2.55])的区分度。如果一本书全篇平淡(真实强度0-2)，会被映射到1.5-3.3，与中等强度书(真实4-6→4.3-5.7)的区分度不足。
**建议**: 使用 calibrate_v2 的 Platt Scaling 公式 `LLM = 0.106*Rule + 2.8` (从 L341注释)，至少这是数据驱动的。

### [#10] LLM截断1200字丢失章末信息？
**判定**: ⚠️
**依据**: `llm_batch_score.py:80` — `chapter_text[:1200]`
**问题**: 网文章节平均3000字。前1200字通常是铺垫/承上，章末高潮和钩子在后1800字。LLM看不到结尾无法准确评估 hook_quality 和 retention。
**影响**: hook评估严重失真。retention(留存力)是"看完后想不想继续"，没看到章末无法判断。
**建议**: 改为取前800字+后400字: `chapter_text[:800] + "\n...[中段]...\n" + chapter_text[-400:]`

### [#11] Qwen3.5-Q4_K_M 的 Self-Preference Bias
**判定**: ⚠️ — 已知风险, 部分缓解
**依据**: arxiv 2410.21819 — LLM系统性偏好低perplexity文本
**评估**: 
- 风险存在: Qwen3.5可能对写法"标准"的章节打高分，对风格独特但有效的章节打低分
- 部分缓解: 只用LLM打6个维度中4个(留存力全LLM，爽点70%LLM，钩子5%LLM，冲突0%LLM)，并非所有维度依赖LLM
- 未缓解: 留存力完全依赖LLM，且LLM bias在retention上无任何校准
**建议**: 用DeepSeek-R1(8002)做交叉验证。两份打分取均值可降低单模型bias。

### [#12] 分层采样是否错过密度集中区？
**判定**: ⚠️
**依据**: `llm_batch_score.py:223-225` — `step = max(1, len(chapters) // max_chapters)`
**问题**: 均匀步长采样。如果一本书前200章日常平淡、后200章高潮密集，60章采样中可能只有15章来自高潮段——低估整本书的爽点密度。
**建议**: 按 rhythm CSV 中的 pleasure_intensity 排序后，取 top-20 + random-20 + bottom-20 进行分层采样。

### [#13] Self-Consistency 温度范围是否过窄？
**判定**: ⚠️
**依据**: `llm_batch_score.py:126` — `temps = [0.1, 0.2, 0.3][:n_samples]`
**问题**: 0.1-0.3 对于 Qwen3.5 的 token distribution 影响极小，3个样本可能高度相似，达不到 self-consistency 的多样性目的。TURN(arxiv 2502.05234) 推荐的是 temperature search，至少涵盖 0.1-0.7。
**建议**: 改为 `[0.1, 0.3, 0.7]`。

### [#14] 分类mode投票: 平局处理？
**判定**: ⚠️
**依据**: `llm_batch_score.py:201-204` — `Counter(r[...]).most_common(1)[0][0]`
**问题**: 3样本各不同(如fast/medium/slow)时，`most_common(1)` 返回第一个遇到的，非确定性。
**建议**: 平局时 fallback 到 `medium`(中性值)或温度最低样本的答案。

### [#15] 序章编号处理
**判定**: ⚠️
**依据**: `rhythm_analyzer.py:224-228` — 序章(num=0)重置为1
**问题**: 序章被编号为1，若书也有"第一章"，两者冲突。`rule_rows.get(ch_num, {})` 会错误匹配。
**建议**: 序章使用特殊编号如 `0` 或 `-1`，不参与编号递增。

### [#16] n=10 的百分位分辨率
**判定**: ❌ — 核心统计问题
**依据**: `genre_synthesizer.py:289` — `(rank-1)/(n-1)*100`
**问题**: n=10时，每档间隔 11.1%。3本书之差才跨过一个档位。举例: 排名第1=0%, 第2=11%, ..., 第10=100%。第4和第5名差11%，这个粒度下无法可靠区分。
**在ML评估中**: n=10, r=0.66 的 95% CI ≈ [0.05, 0.91] (Fisher z-transform)，几乎不提供统计显著性。
**建议**: 纳入非末世书的base指标(如hook/conflict基本维度)，扩展n到30+。

### [#17] _p() 对小样本的线性插值
**判定**: ✅ (在约束条件下合理)
**依据**: `genre_synthesizer.py:293-301` — 线性插值百分位
**评价**: n=10时用线性插值比阶梯函数好。已知局限(每档间隔大)而非方法错误。

### [#18] r²/(1+r²) 的方差假设
**判定**: ⚠️ — 隐含假设未验证
**依据**: `genre_synthesizer.py:50-55`
**公式推导**: `w_rule = r²/(1+r²)` 来自 `w = σ²_llm/(σ²_rule + σ²_llm)` 且 `σ²_rule/σ²_llm = 1/r² - 1`，即假设误差方差比 = 1/r²。
**问题**: 如果有100章校准数据，可以实际计算 `σ²_rule` 和 `σ²_llm` 而非假设。实际 LLM 的方差可能更大（采样噪声 + 提示敏感）。
**建议**: 从 calibrate_v2 数据计算实际 `Var(Rule_intensity - LLM_intensity)`，用经验 shrinkage。

### [#19] pos_density 缩放假设
**判定**: ⚠️
**依据**: `genre_synthesizer.py:445` — `(opening_pos_density + 0.5) * 10.0`
**问题**: 假设 pos_density 范围[-0.5, 0.5]。实测火书中开篇pos_density通常在[-0.1, 0.4]。但自己写的小说可能突破此范围。clamp 到 [0,10] 会削平极端值。
**建议**: 使用火书池的 P5 和 P95 做 min-max 归一化，而非固定常数。

### [#20] Grid Search 权重是否过拟合？
**判定**: ⚠️ — 高风险
**依据**: `genre_synthesizer.py:497-510` — 4组子类型权重
**问题**: Grid Search 在10本书上调了4组×3参数=12个权重，但只有10个数据点。这几乎是 "每个数据点一个参数"，严重过拟合风险。Spearman 0.66 如果用同一批书调参+评估，是 inflated estimate。
**建议**: (1) 用 leave-one-out 验证真实泛化r。(2) 改用2组而非4组权重(打脸+智斗合并, 通用+羁绊合并)减少参数。(3) 优先用简单线性组合而非分段权重。

### [#21] "前3章钩子"含义混淆
**判定**: ⚠️
**依据**: `genre_synthesizer.py:447-455`
**问题**: "前3章钩子"用了 ch3 的 `hook_density` 均值(全章范围)，但 hook_type 检查的是章末500字(L292)。这两个是不同的东西——一个是全章钩子词密度，一个是章末悬念类型。命名和实际含义不一致。
**建议**: 重命名为"开篇钩子密度"，并在文档中明确说明与章末钩子的区别。

### [#22] 留存力未百分位归一化
**判定**: ❌ — 权重失衡
**依据**: `genre_synthesizer.py:479` — `scores["读者留存力"] = round(llm_avg_retention * 10)`
**问题**: 其他维度都是0-100百分位排名，唯独留存力是 LLM评分×10(范围10-100)。如果某书LLM retention=7.0，留存力=70；但如果火书池LLM retention中位数是6.0，70分在百分位中本应是~80分——**低估了10分**。反之亦然。
**建议**: `scores["读者留存力"] = percentile_score(llm_avg_retention, pool, "retention")` 并对 pool 添加 retention 维度。

### [#23] Survival 留存公式完全 heuristic
**判定**: ❌
**依据**: `genre_synthesizer.py:743-749` — `est = 85 - ratio*40 - max(0,0.3-conflict)*30 + ...`
**问题**: 85/40/30/10/2 — 全部无来源。这不是数据分析，是"看起来合理"的猜测。如果用于向作者展示"你的第3段留存预估70%"，这个数字可能是误导性的。
**建议**: 要么标注 "仅做趋势参考，非精确预估"，要么从火书实际完读率数据做回归(literally impossible without per-segment 完读率 data)。作为替代：只输出风险等级(🟢/🟡/🔴)基于零钩子比例，不做具体百分比。

### [#24] _find_book_stem() 匹配脆弱性
**判定**: ⚠️
**依据**: `genre_synthesizer.py:93-102` — 前5行章节号匹配
**问题**: 如果两本书的第1-5章都有LLM数据且章节号相同，匹配非确定性。虽然在独立分析单本书时正确，但批量跑时可能错配。
**建议**: 从 `analyze_single_novel()` 传入 csv_name 的 stem，避免模糊匹配。

### [#25] rhythm CSV 和 LLM CSV 章节号对齐
**判定**: ⚠️
**依据**: 两个CSV都通过 `extract_chapters()` 产生章节号
**问题**: 但如果 TXT 文件被修改(如转码后空白变化)，两次提取结果可能不同。章节号一旦错位，LLM分数会融合到错误的章上。
**建议**: `llm_batch_score.py` 写入CSV时也保存章节标题(`ch["title"]`)，融合时用标题做二次验证。

### [#26] n=10, r=0.66 的统计显著性
**判定**: ❌ — 置信区间极宽
**计算**: Fisher z = arctanh(0.66) = 0.793, SE = 1/√(10-3) = 0.378, 95%CI = [tanh(0.793±1.96×0.378)] = [tanh(0.052), tanh(1.534)] = [0.05, 0.91]
**结论**: 无法以95%置信度拒绝"真实相关r=0"的假设。n=10 远不够。
**建议**: 最低 n=20 (SE=0.243, 95%CI≈[0.25,0.87]) 才有基本区分力。目标 n=30 可达合理精度。

### [#27] 数据泄露: 完读率用于调权又用于评估
**判定**: ❌ — 方法论问题
**依据**: Grid Search 用完读率调4组权重(L497-510)，然后报告 Spearman r vs 完读率
**问题**: 同一批数据既调参又评估 → r 被高估。正确的做法是留一法(LOOCV)或划分调参集/测试集。
**建议**: 实现 LOOCV: 每次留1本书做测试，其余9本调权重，报告10次测试的平均r。

### [#28] compute_commercial_score() 崩溃路径
**判定**: ✅ (防御充分)
**检查**: 早退条件 L378 `total<10`, L383 `pool["n_books"]<3`, LLM fallback L432-434, all divisions guarded by max(denom,1)
**评价**: 防御编程到位，无已知崩溃路径。

### [#29] auto_benchmark() 路径构建
**判定**: ✅
**依据**: `genre_synthesizer.py:1057-1060` — `output_dir = DATA_DIR / "analysis" / genre / "synthesis"`, DATA_DIR = PROJECT_ROOT / "data"
**评价**: 路径由 PROJECT_ROOT 派生，不会漂移。但如果 genre 含特殊字符(如路径分隔符)，可能创建非预期目录。当前 genre="末世"无此风险。

## 优化方向评价

### 方向A: Platt Scaling 替代手动调参
**可行性**: ⭐⭐⭐⭐⭐ 强烈推荐
**理由**: calibrate_v2 已有100章数据，线性回归 `LLM = a*Rule + b` 可直接计算。L341注释已有系数(0.106*Rule+2.8)，只是未应用。替换 `*0.7+1.5` 为零成本提升。
**实现工作量**: 低（改1行公式）

### 方向B: TF-IDF 替代纯正则
**可行性**: ⭐⭐⭐ 中长期有价值，短期不紧迫
**理由**: 正则对变体(同义词/错别字)不敏感，TF-IDF+种子词扩展可提升覆盖率15-25%。但需要训练语料和验证，建议作为 P1 而非 P0。
**实现工作量**: 中（需新增模块+回归测试）

### 方向C: DeepSeek-R1 双模型交叉验证
**可行性**: ⭐⭐⭐⭐ 推荐在打分环节执行
**理由**: 对单模型bias(特别是Self-Preference)有实质性抵消。成本是2倍LLM调用时间。建议对关键章节(top/bottom 10%)做双模型验证，而非全量。
**实现工作量**: 低（复用已有 llm_score_rubric + 改 target URL）

### 方向D: n=10→30 扩展火书池
**可行性**: ⭐⭐⭐⭐⭐ 最优先
**理由**: 这是改善统计可靠性的根本手段——所有百分位排名、Spearman r、Grid Search 的精度都受n限制。如果7本非末世书的基础指标(hook/conflict)与末世兼容，纳入后n=17→CI宽度缩减~25%。
**实现工作量**: 中（需验证跨题材兼容性 + 跑7×60=420章LLM打分）
