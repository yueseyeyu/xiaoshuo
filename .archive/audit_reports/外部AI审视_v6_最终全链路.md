# 外部AI审视指令 — 末世小说商业评分 v8 最终全链路

> 请以「独立 AI 审计员」身份审查以下系统。逐条判定 ✅/⚠️/❌ + 行号依据 + 建议。

---

## 系统概述

对 10 本末世火书（14,696章）进行三层分析，输出商业签约评分。

### 全链路

```
第1层: rhythm_analyzer.py v8 (正则规则引擎)
  └─ 章节提取(4层fallback) → 9类爽点(r加权) + 5类冲突 → 22维CSV
第2层: llm_batch_score.py v8 (LLM量规打分, 并行x2)
  └─ 首尾截取(600+600字) → Qwen3.5-9B 6维量规评分 → LLM CSV
第3层: genre_synthesizer.py v8 (融合+拆书+商业评分)
  └─ Bayesian BMA融合 + 百分位排名 + 留存力归一化 + LOOCV → 报告
```

### 基础设施
- 模型: Qwen3.5-9B Q4_K_M (5.68GB), RTX 5060 8GB
- 服务器: --parallel 2, HTTP连接复用, ThreadPoolExecutor 2线程
- 速度: 600章 ~25min (优化前~50min)

## 第1层: rhythm_analyzer.py v8 审查

### 1.1 章节提取
**代码**: `extract_chapters()` L146-183
**审查问题**:
1. 4层fallback: 第X章→序章+章X→数字标题→裸数字。是否仍需第5层("卷第X章"、英文Chapter)?
2. L193 `len(pure_text) < 50: continue` — 序章/短章是否被错误丢弃?

### 1.2 爽点检测 (v8: r加权聚合)
**代码**: L267-278
```python
weighted_pleasure = (
    slap_count * 0.096 + level_count * 0.086 + crush_count * 0.300 +
    comeback_count * 0.178 + hidden_count * 0.108 + general_count * 0.108 +
    bond_count * 0.155 + cognitive_count * 0.017 + sacrifice_count * 0.0 +
    physio_count * 0.179
)
pos_density = total_pleasure / max(wc, 1) * 100
```
**审查问题**:
3. `hidden_count` 和 `general_count` 用 0.108(估算) — calibrate_v2 未测这两个特征。估算权重是否安全?
4. `sacrifice_count * 0.0` (r=-0.13→0) — 负相关特征直接置零，但牺牲场景本身就是负情绪，可能与爽点正交而非反向。是否应保留0而非负数?
5. 羁绊消歧 L264: 硬编码人名已移除，但 `"你|我|他|她|眼中|心里|轻声|沉默|握住|凝视"` 是否足够通用?

### 1.3 冲突检测 (v8: 无变更)
**代码**: CONFLICT_KW_ALL L84-95 + PLEASURE_BOND L37-41 (移除战斗重叠词)
**审查问题**:
6. BOND正则移除 `"挡在|护在|扶住|接住"` 减少战斗误判 — 是否同时丢掉了真正的羁绊保护场景?
7. 冲突合并5类(战斗+心理+道德+环境+社会) — 5类权重是否均等? 心理博弈类冲突在末世文中是否比战斗冲突更重要?

### 1.4 爽点强度公式 (v8: Platt Scaling回退)
**代码**: L337-350
```python
pleasure_raw = (
    pos_density * 2.0 + conflict_density * 1.5 + excl_density * 0.5 +
    hook_density * 0.5 - neg_density * 0.2 +
    physio_count * 2.0 / max(wc/100, 1)
)
pleasure_raw = pleasure_raw * 0.7 + 1.5  # holdover until recalibration
```
**审查问题**:
8. `dialogue_ratio` 已移除(原r=-0.091) — 是否还有遗漏的噪声特征?
9. `*0.7+1.5` 是 v5旧公式，v8修改后需重校准。当前低分压缩(+1.5偏移)导致[0,2]区间不可达。是否应在重校准前使用更保守的 `max(0, pleasure_raw)`?

## 第2层: llm_batch_score.py v8 审查

### 2.1 首尾截取策略
**代码**: `_build_rubric_prompt()` L78-88
```python
if len(text) > 1200:
    text = text[:600] + "\n...[中段省略]...\n" + text[-600:]
```
**审查问题**:
10. 对于800字的短章，`text[:600] + text[-600:]` 会产生前后重叠。`len(text) > 1200` 守卫正确但有边界情况: 1201字章会触发截取，1199字章用全文。这个拐点是否合理?
11. `...[中段省略]...` 提示语是否影响LLM评分行为?

### 2.2 并行化设计 (v8: ThreadPoolExecutor x2)
**代码**: L245-275
```python
def _score_chapter(packed):
    t_conn = http.client.HTTPConnection(LLAMA_HOST, timeout=45)
    try: ... finally: t_conn.close()

with ThreadPoolExecutor(max_workers=2) as executor:
```

**审查问题**:
12. 每线程独立创建HTTP连接 — 是否造成不必要的TCP握手开销? 连接池(2个连接复用)是否更优?
13. `fut.result()` 如果 `_score_chapter` 内部抛异常(如 chapters[idx] 索引错误)，异常会在此处传播并终止executor。是否需要 `try/except` 包装?
14. `ordered[i]` 写入是线程安全的(每个i唯一)，但 `llm_score_rubric` 内部的 `llm_score_self_consistency` 是否完全线程安全?

### 2.3 评分量规
**审查问题**:
15. 6维评分: 爽点强度/冲突/情绪/节奏/钩子/留存力。每个维度等级有锚定描述。这个量规是否覆盖了末世文的独特性(如"资源短缺的焦虑感"、"道德困境中的抉择" 等)?

## 第3层: genre_synthesizer.py v8 审查

### 3.1 Bayesian BMA权重
**代码**: `_load_bayesian_weights()` L27-71
```python
r2 = max(r, 0.01) ** 2
w_rule = max(0.05, r2 / (1.0 + r2))
```
**审查问题**:
17. `r²/(1+r²)` 假设 rule和LLM的误差方差相等。从100章校准数据，实际 `var(LLM)/var(rule) ≈ ?` 是否需要引入经验方差比?
18. `w_rule` 最低 0.05 防止LLM垄断 — 对于 r=-0.13 的特征(牺牲)，这个下限是否合理? 还是应该 `max(0, r²/(1+r²))` 允许为0?

### 3.2 留存力百分位归一化 (v8新增)
**代码**: L502 + 火书池新增 `retention` 维度
```python
scores["读者留存力"] = percentile_score(llm_avg_retention, pool, "retention")
```
**审查问题**:
19. retention 池从LLM CSV读取每本书的平均retention。但 `_load_all_llm_scores()` 缓存了全量LLM数据，`get_firebook_pool()` 另读了LLM CSV。两次IO — 是否有缓存不一致风险?
20. retention百分位在 pool 中仅有 `_sorted` 和 `p50`，缺少 `p25/p75`。`percentile_score()` 只用 `_sorted` 做秩排名 — 对于 retention 这个指标，秩排名是否足够?

### 3.3 商业评分公式
**代码**: L504-518
```python
sign = (scores["前3章钩子"] + scores["前3章冲突"] + scores["首章爽点"]) / 3
retain_old = (scores["零钩子连续"] + scores["打脸频率"] + scores["爽点多样性"]) / 3
bonus = mean(bonus_dims)  # 反转频率+悬念频率+大爽频率+留存力
overall = sign * gw["sign"] + retain_old * gw["retain"] + bonus * gw["bonus"]
```
**审查问题**:
21. Grid Search 子类型权重(Spearman 0.66, n=10) — LOOCV 当前为近似实现(未真正剔除held-out书)。真LOOCV的Spearman预期多少?
22. `bonus_dims` 包含4个维度等权平均后又与外层 `gw["bonus"]` 相乘 — 这是"加权中的加权"。是否存在维度权重二次分配问题?

### 3.4 Survival分段留存
